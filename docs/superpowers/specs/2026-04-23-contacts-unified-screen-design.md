# Contacts Unified Screen Design

## Goal

Объединить текущие пользовательские сценарии `Friends` и `QR` в один общий экран `Контакты`, чтобы добавление друзей, показ собственного QR и разбор входящего QR происходили в одном месте без прыжков между вкладками.

Date: `2026-04-23`

## Context

Сейчас в приложении есть две отдельные вкладки:

- `/app/friends` для социального графа, инвайтов и активности;
- `/app/qr` для собственного QR, ручного resolver-сценария и переходов.

По пользовательскому смыслу это один поток:

- показать свой QR;
- отсканировать или вставить чужой QR;
- добавить друга;
- подтвердить связь;
- увидеть состояние контактов.

Разделение на два экрана делает этот сценарий искусственно разорванным.

## Goals

- Сделать одну пользовательскую вкладку `Контакты` вместо двух отдельных `Друзья` и `QR`.
- Поместить на один экран:
  - собственный QR пользователя;
  - входящий QR / resolver;
  - ручное добавление по `user_id`;
  - входящие и исходящие инвайты;
  - активные контакты;
  - социальную активность.
- Сохранить текущие backend API и существующие данные без новой серверной модели.
- Оставить старые маршруты рабочими через редирект, чтобы не ломать ссылки и переходы.

## Non-Goals

- Не добавлять камеру или настоящее сканирование с устройства в этой итерации.
- Не проектировать новый backend для контактов.
- Не переделывать AI-модуль в рамках этой задачи.
- Не строить сложную вкладочную навигацию внутри нового экрана, если можно обойтись единым scroll-first layout.

## Chosen Approach

Выбран подход `single unified route`.

Суть:

- в основной навигации остается одна вкладка `Контакты`;
- она открывает единый экран, собирающий функциональность старых `FriendsPage` и `QrPage`;
- маршруты `/app/friends` и `/app/qr` становятся редиректами на `/app/contacts`;
- старые отдельные страницы больше не используются как самостоятельные пользовательские поверхности.

Это лучший вариант, потому что он решает именно продуктовую проблему: пользователь мыслит не “друзья отдельно, QR отдельно”, а “контакты и обмен доступом”.

## Information Architecture

### Navigation

В `appLinks`:

- удалить отдельные пункты `Друзья` и `QR`;
- добавить один пункт `Контакты` с маршрутом `/app/contacts`.

В мобильной нижней навигации:

- заменить текущие отдельные кнопки `Друзья` и `QR` одной кнопкой `Контакты`.

### Routes

Новая основная точка входа:

- `/app/contacts`

Маршруты совместимости:

- `/app/friends` -> редирект на `/app/contacts`
- `/app/qr` -> редирект на `/app/contacts`

## Screen Structure

Экран `Контакты` должен быть одним длинным экраном, а не набором внутренних табов.

### Hero block

Верхний блок объясняет единый смысл раздела:

- это центр контактов и QR-взаимодействия;
- здесь можно показать свой код, принять чужой и управлять кругом контактов.

Hero-метрики:

- активные контакты;
- открытые инвайты;
- входящие запросы;
- количество событий активности.

### Section 1: My QR

Первый крупный блок:

- показать мой QR / payload;
- дать действие обновления;
- оставить кнопку “использовать в resolver”;
- оставить краткую расшифровку CTA.

Этот блок должен ощущаться как способ “дать себя добавить”.

### Section 2: Scan Or Paste QR

Второй блок:

- textarea / input для входящего payload;
- разбор и валидность;
- объяснение результата;
- CTA следующего шага.

Этот блок должен ощущаться как способ “принять чужой QR”.

### Section 3: Manual Add

Третий блок:

- добавление по `user_id`;
- текущий ручной invite flow;
- короткая подсказка, когда использовать ручной сценарий вместо QR.

### Section 4: Incoming And Outgoing

Дальше:

- входящие инвайты;
- исходящие инвайты;
- действия подтверждения;
- статусы ожидания.

### Section 5: Active Contacts

Отдельный блок:

- принятые связи;
- базовая информация по контакту;
- дата / источник.

### Section 6: Activity

Последний блок:

- социальная лента активности;
- пустое состояние;
- ошибки сети.

## Data And Component Strategy

Новый экран не должен переписывать backend-логику.

Рекомендуемая реализация:

- создать новый экран `ContactsPage`;
- вынести блоки из `FriendsPage` и `QrPage` в более мелкие UI-секции или переиспользовать текущую разметку с минимальным переносом;
- сохранить существующие запросы:
  - `getFriends`
  - `getFriendActivity`
  - `getMyQr`
  - `resolveQr`
  - `inviteToFriends`
  - `acceptFriendInvite`
- не менять контракт API ради этого объединения.

## Redirect And Compatibility Rules

- Любая существующая кнопка или ссылка на `/app/friends` должна продолжать работать.
- Любая существующая кнопка или ссылка на `/app/qr` должна продолжать работать.
- Обе точки должны переводить пользователя на `/app/contacts`.
- Внутренние CTA в `AI`, `Friends`, `QR` и связанных местах нужно обновить на `Контакты`, где это уже чисто пользовательский copy.

## UI Copy

Новая терминология:

- вкладка: `Контакты`
- hero смысл: единый центр для QR и связей
- CTA:
  - `Открыть контакты`
  - `Показать мой QR`
  - `Вставить QR`
  - `Добавить по ID`

Старые тексты `Открыть друзей` и `Открыть QR` должны быть убраны или заменены на нейтральные переходы в `Контакты`.

## Testing Strategy

Проверить:

- в навигации есть `Контакты`, а отдельных пользовательских пунктов `Друзья` и `QR` больше нет;
- `/app/contacts` рендерит объединенный экран;
- `/app/friends` и `/app/qr` редиректят на `/app/contacts`;
- на объединенном экране присутствуют:
  - блок собственного QR;
  - блок resolver;
  - ручной инвайт;
  - входящие / исходящие / активные связи;
  - активность;
- существующие тесты маршрутизации и ключевых страниц обновлены под новую архитектуру.

## Risks

- если перенести оба экрана в один файл без декомпозиции, `ContactsPage` быстро станет перегруженным;
- можно случайно оставить старый copy и старые кнопки, из-за чего экран будет выглядеть как механическая склейка;
- часть старых ссылок по приложению может продолжать называть экран `Друзья` или `QR`, даже если маршрут уже общий.

## Implementation Boundaries

В рамках задачи:

- навигация;
- роутинг;
- единый пользовательский экран `Контакты`;
- обновление CTA и видимых подписей.

Вне рамок:

- камера-сканер;
- новые backend endpoints;
- переработка AI-модуля;
- новая модель ролей или прав доступа.
